事务和锁 - MySQL

1. 事务概述

1.1 四大特性

事务具有四大特性（ACID）：

  • 原子性：事务是不可分割的最小操作单元，要么全部成功，要么全部失败
  • 一致性：事务完成时，必须使所有的数据都保持一致状态
  • 隔离性：数据库系统提供的隔离机制，保证事务在不受外部并发操作影响的独立环境下运行
  • 持久性：事务一旦提交或回滚，它对数据库中的数据的改变就是永久的

1.2 基本操作

语法

事务是一组操作的集合，事务会把所有的操作作为一个整体一起向系统提交或撤销操作请求，即这些操作要么同时成功，要么同时失败。

  • 查看事务提交方式

    select @@autocommit;

  • 设置事务提交方式：默认为1，即自动提交

    set @@autocommit = 0;

  • 开启事务

    start transaction; 或 begin;

  • 提交事务：将所有的DML语句操作历史记录和底层硬盘数据来一次同步

    commit;

  • 回滚事务：将所有的DML语句操作历史记录全部清空

    rollback;

2. 并发事务

2.1 并发事务问题

  • 脏读：一个事务读到另外一个事务还没有提交的数据

  • 不可重复读：一个事务先后读取同一条记录，但两次读取的数据不同，称之为不可重复度

  • 幻读：一个事务按照条件查询数据时，没有对应的数据行，但在插入数据时，又发现这行数据已经存在。

2.2 事务隔离级别

隔离级别

  • 读未提交 (read uncommitted)： 一个事务可以读取另一个未提交事务的数据
  • 读提交 (read committed)：一个事务要等另一个事务提交后才能读取数据
  • 可重复读 (repeatable read)：在开始读取数据（事务开启）时，不再允许修改操作
  • 串行 (serializable)：是最高的事务隔离级别，事务串行化顺序执行，但性能低且效率低下
隔离级别 脏读 不可重复读 幻读
read uncommitted 未解决 未解决 未解决
read committed 解决 未解决 未解决
repeatable read（默认） 解决 解决 未解决
serializable 解决 解决 解决

语法

  • 查看事务隔离级别

    select @@transaction_isolation;

  • 设置事务隔离级别

    set [session|global] transaction isolation level 隔离级别;

3. 锁分类

按照锁的粒度分类

  • 全局锁：锁定数据库中的所有表
  • 表级锁：每次操作锁住整张表
  • 行级锁：每次操作锁住对应的行数据

不同存储引擎支持情况如下：

存储引擎 表级锁 行级锁
MyISAM 支持 不支持
InnoDB 支持 支持
MEMORY 支持 不支持
BDB 支持 不支持

按照操作类型分类

  • 读锁（共享锁）：针对同一份数据，多个读操作可以同时进行而不会互相影响。
  • 写锁（排它锁）：当前操作没有完成之前，它会阻断其他写锁和读锁。

4. 全局锁

语法

全局锁就是对整个数据库实例加锁，后续的 DML 语句， DDL 语句，以及更新操作的事务提交语句都将被阻塞，直至全局锁被释放后再执行。全局锁常用于做全库的逻辑备份。

  • 加全局锁

    flush tables with read lock;

  • 解全局锁

    unlock tables;

  • 数据库备份：非 SQL 语句，而是一个 MySQL 的工具，直接在命令行窗口执行

    mysqldump -h主机地址 -u用户名 -p用户密码 数据库名 > 路径/数据库名.sql;

    在 InnoDB 引擎中可以在备份时在 mysqldump 后加上参数 --single-transaction 完成不加锁的一致性数据备份。

5. 表级锁

每次操作锁住对应的表数据。应用在 MyISAM、InnoDB、BDB 等存储引擎中。

5.1 表锁

概述

  • 表共享锁（read lock）：又名读锁，所有客户端均只可读、不可写。上锁客户端执行写会报错，其他客户端执行写会阻塞直至释放锁。

  • 表排他锁（write lock）：又名写锁，本客户端可读可写，其余客户端均不可读不可写

语法

  • 加锁

    lock tables 表名列表 read/write;

  • 释放锁

    unlock tables; 或 客户端断开连接

5.2 元数据锁

概述

在 MySQL5.5 中引入了元数据锁（MDL），加锁过程是系统自动控制，在访问一张表的时候会自动加上，避免 DML 与 DDL 冲突。

查看元数据锁可以使用如下查询命令：

select object_type,object_schema,object_name,lock_type,lock_duration from performance_schema.metadata_locks;

注意

  • 当对一张表进行增删改查时会自动添加 MDL 共享锁（读锁），当对表结构进行变更操作的时候会自动添加 MDL 排他锁（写锁）。
  • 开启事务后才能自动加上元数据锁，提交事务后自动释放元数据锁。若为自动提交，则这个过程速度很快几乎看不出来。

添加锁

以下事务中的操作会自动添加元数据锁：

对应 SQL 自动添加的MDL 说明
lock tables 表名列表 read/write shared_read_only/shared_no_read_write
select、select ... lock in share mode shared_read 与 shared_read、shared_write 兼容，与 exclusive 互斥
insert、update、delet、select ... for update shared_write 与 shared_read、shared_write 兼容，与 exclusive 互斥
alter table exclusive 与其他 MDL 都互斥

5.3 意向锁

概述

为了避免加的表锁与行锁冲突，在 InnoDB 中引入了意向锁，使得表锁不用检查每行数据是否加锁，直接检查意向锁。

意向锁分为两类：

  • 意向共享锁（IS）：与意向锁、表锁共享锁（read）兼容，与表锁排他锁（write）互斥
  • 意向排他锁（IX）：与意向锁兼容，与表锁共享锁（read）、表锁排他锁（write）互斥

查看意向锁和行锁可以使用如下查询命令：

select object_schema,object_name,index_name,lock_type,lock_mode,lock_data from performance_schema.data_locks;

注意

  • 当对一张表添加共享行锁时会自动添加意向共享锁（IS），当对一张表进行增删改查、或添加排他行锁时会自动添加意向排他锁（IX）
  • 开启事务后才能自动加上意向锁，提交事务后才会自动释放意向锁。若为自动提交，则这个过程速度很快几乎看不出来。

添加共享锁

事务中 select … lock in share mode 会添加意向共享锁（IS）。

添加排他锁

事务中 insert、update、delete、select … for update 会添加意向排他锁（IX）。

6. 行级锁

每次操作锁住对应的行数据。应用在 InnoDB 存储引擎中。

InnoDB 的行级锁时通过对索引上的索引项加锁来实现的，而不是对记录加锁。

6.1 行锁

概述

锁定单个行记录的锁，防止其他事务对此行进行 update 和 delete。在 RC、RR 隔离级别下都支持。

行锁分为两类：

  • 共享锁（S）：允许一个事务去读一行，阻止其他事务获取相同数据集的排他锁。即共享锁与共享锁兼容，与排他锁互斥。
  • 排他锁（X）：允许获取排他锁的事务更新数据，阻止其他事务获取相同数据集的共享锁和排他锁。排他锁与共享锁和排他锁互斥。

查看意向锁和行锁可以使用如下查询命令：

select object_schema,object_name,index_name,lock_type,lock_mode,lock_data from performance_schema.data_locks;

注意

  • InnoDB 的行锁是针对于索引加的锁，不通过索引条件检索数据，那么 InnoDB 将对表中所有的记录加锁，锁的类型取决于语句原本的锁类型

  • 针对唯一索引进行检索时，对已存在的记录进行等值匹配时，将会自动优化为行锁。

  • 开启事务后才能自动加上行锁，提交事务后才会自动释放行锁。若为自动提交，则这个过程速度很快几乎看不出来。

添加锁

以下事务中的操作会添加行锁：

对应 SQL 行锁类型 说明
insert 排他锁（X） 自动加锁
update 排他锁（X） 自动加锁
delete 排他锁（X） 自动加锁
select 不加任何锁
select ... lock in share mode 共享锁（S） 需要手动在 select 后面加 lock in share mode
select ... for update 排他锁（X） 需要手动在 select 后面加 for update

6.2 间隙锁和临键锁

概述

  • 间隙锁：锁定索引记录到前一个记录之间的间隙（不含该记录），防止其他事务在这个间隙进行 insert，产生幻读。在 RR 隔离级别下支持。

  • 临键锁：行锁和间隙锁的组合，同时锁住数据，并锁住数据前面的间隙。在 RR 隔离级别下支持。

注意

  • 默认情况下，InnoDB 在 RR 事务隔离级别运行，InnoDB 使用 next-key 锁进行搜索和索引扫描，以防止幻读。
  • 间隙锁唯一的目的时防止其他事务插入间隙。间隙锁可以共存，一个事务采用的间隙锁不会阻止另一个事务在同一间隙上采用间隙锁。
  • 开启事务后才能自动加上间隙锁，提交事务后才会自动释放间隙锁。若为自动提交，则这个过程速度很快几乎看不出来。

添加锁

以下事务中的操作会添加间隙锁：

  • 索引上的等值查询（唯一索引）：给不存在的记录加行锁时，优化为间隙锁。

  • 索引上的等值查询（普通索引）：给不存在的记录加行锁时，会访问到不满足条件的的值，next-key lock 退化为间隙锁

  • 索引上的等值查询（普通索引）：给已存在的数据添加行锁时，会给该数据加上临键锁，给该数据之后的间隙加上间隙锁。

  • 索引上的范围查询（唯一索引）：会访问到不满足条件的第一个值为止，并将该值及该值之后的所有数据全部加上临键锁（包括正无穷）
